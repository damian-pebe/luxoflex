# Alembic in this project

Alembic is the **database migration** tool used by this backend. It lets you version schema changes (tables, indexes, FKs, etc.) and apply or revert them in a controlled way in each environment.

---

## What Alembic does

- **Revisions (migrations):** Python files under `alembic/versions/` that define:
  - **`upgrade()`** — applies the change (create table, add column, etc.).
  - **`downgrade()`** — reverts the change.
- **History:** the database stores in the `alembic_version` table which revision is applied, so the tool knows which migrations still need to run.
- **Autogenerate:** it can compare SQLAlchemy models with the current database and generate the migration code for you (you then review and adjust it).

In this project the **database URL** is taken from the app config (`DATABASE_URL` in `.env`), not from `alembic.ini`, so the same env-based URL is used in development, staging, and production.

---

## Repo structure

```
alembic/
├── env.py              # Config: uses settings.database_url, model metadata, async mode
├── script.py.mako      # Template for new revision files
└── versions/           # Migrations (revisions)
    └── xxxxx_descriptive_name.py
alembic.ini             # Alembic config (script location, logging, etc.)
```

- **`env.py`** imports models from `src.models` so `Base.metadata` has all tables and Alembic can autogenerate and run migrations.
- **`alembic.ini`** sets `script_location = alembic` and logging options; the actual URL is overridden in `env.py` with `settings.database_url`.

---

## How it runs (online mode)

- Alembic uses the **same `DATABASE_URL`** as the backend (loaded by `src.core.config.settings` from `.env`).
- Migrations run in **online** mode with an **async** engine (`async_engine_from_config` + `run_sync`), compatible with `postgresql+asyncpg://...`.

Some tables are **excluded** from autogenerate and comparison so we don’t touch schema managed outside the ORM (e.g. PostGIS or legacy tables). In `env.py`:

```python
EXCLUDE_TABLES = {"spatial_ref_sys", "vehicles", "vehicle_events", "alembic_version"}
```

Those tables will not be created or altered by autogenerated migrations.

## Useful commands

Run commands from the **project root** (where `alembic.ini` lives). With `uv`:

```bash
uv run alembic <command>
```

### Show current state

```bash
uv run alembic current
```

Shows the revision currently applied in the database (from `alembic_version`).

### Show revision history

```bash
uv run alembic history
```

Lists revisions and their `down_revision` chain (which follows which).

### Apply all pending migrations (upgrade)

```bash
uv run alembic upgrade head
```

Applies all revisions in order up to `head`. This is the usual command when deploying or setting up a new environment.

### Upgrade to a specific revision

```bash
uv run alembic upgrade <revision_id>
```

Example: `uv run alembic upgrade dfaeab2e75a7`.

### Revert a migration (downgrade)

```bash
uv run alembic downgrade -1
```

Moves back one revision. To go back to a specific revision:

```bash
uv run alembic downgrade <revision_id>
```

### Create a new revision (migration)

**Autogenerate (recommended):** compare models with the database and generate `upgrade`/`downgrade`:

```bash
uv run alembic revision --autogenerate -m "description of the change"
```

A new file is created under `alembic/versions/` with a unique ID and your message. **Always review the generated code** before running `upgrade` (index names, FKs, excluded tables, etc.).

**Empty revision (manual):** if you want to write the migration by hand:

```bash
uv run alembic revision -m "description of the change"
```

Then edit the `upgrade()` and `downgrade()` functions in the generated file.

## Typical revision file contents

Each file in `alembic/versions/` has:

- **`revision`** — unique ID for this revision.
- **`down_revision`** — ID of the previous revision (or `None` for the first).
- **`upgrade()`** — applies the change (e.g. `op.create_table`, `op.add_column`, `op.create_index`).
- **`downgrade()`** — undoes the change (e.g. `op.drop_table`, `op.drop_column`).

Example snippet generated by autogenerate:

```python
def upgrade() -> None:
    op.create_index(op.f('ix_users_email'), 'users', ['email'], unique=True)

def downgrade() -> None:
    op.drop_index(op.f('ix_users_email'), table_name='users')
```

Naming for tables, columns, and indexes should follow [databaseConventions.md](./databaseConventions.md).

## Recommended workflow

1. **Change models** in `src/models/` (new tables, columns, indexes).
2. **Generate the migration:**  
   `uv run alembic revision --autogenerate -m "add_phone_to_users"`.
3. **Review and fix** the file in `alembic/versions/` (remove excluded tables, adjust names, defaults, etc.).
4. **Apply in your environment:**  
   `uv run alembic upgrade head`.
5. In **other environments** (staging/production) run `alembic upgrade head` with the corresponding `DATABASE_URL`.

To roll back in development you can use `alembic downgrade -1` (or to a specific revision); note that this may cause data loss.

## Troubleshooting

- **“Can’t locate revision”:** the database has a revision that doesn’t exist in `alembic/versions/`. Make sure you have all migration files (e.g. pull from repo) or, in exceptional cases, set the revision manually in `alembic_version`.
- **Autogenerate suggests changes to excluded tables:** don’t use that part; edit the revision and remove operations on `EXCLUDE_TABLES` (or add the table to `EXCLUDE_TABLES` in `env.py` if it should always be excluded).
- **Connection error:** check that `DATABASE_URL` in `.env` is correct and that the database exists and accepts connections (PostgreSQL service running, firewall, etc.).

For more detail, see the [Alembic Tutorial](https://alembic.sqlalchemy.org/en/latest/tutorial.html).
