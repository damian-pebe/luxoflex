# Project Folder Structure

This document describes the folder structure and organization patterns used in the Medadvisor Backend project.

## Overview

The project follows a **modular domain-driven architecture** where each feature/domain is organized as a self-contained module with its own router, schemas, and services.

## Root Directory Structure

```
dx-plus-backend/
├── main.py                 # FastAPI application entry point
├── scripts.py              # CLI scripts for running the server
├── Dockerfile              # Docker container configuration
├── deploy.sh               # Deployment script
├── pyproject.toml          # uv dependencies and project config
├── uv.lock                 # Locked dependency versions (generated by uv)
├── README.md               # Main project documentation
├── docs/                   # Documentation files
└── src/                    # Source code (main package)
```

## Domain Module Structure

Each domain/feature module follows a consistent structure:

```
[domain_name]/
├── __init__.py             # Module initialization
├── [domain_name].py        # FastAPI router with endpoint definitions
├── schemas/                # Pydantic models for this domain
│   ├── __init__.py
│   └── interfaces.py       # Request/Response schemas
└── services/               # Business logic layer
    ├── __init__.py
    └── [service_name].py   # Individual service functions
```

### Example: Appointment Module

```
appointment/
├── __init__.py
├── appointment.py          # Router with endpoints like:
│                           # - POST /appointment
│                           # - PUT /appointment/accept
│                           # - GET /appointment/professionals
├── schemas/
│   └── interfaces.py       # AppointmentRequest, AppointmentResponse, etc.
└── services/
    ├── create_appointment.py
    ├── accept_appointment.py
    ├── reject_appointment.py
    ├── get_appointments.py
    ├── get_appointment_details.py
    ├── get_pending_appointments.py
    ├── get_professionals.py
    ├── get_professional_statistics.py
    ├── appointment_service.py
    └── update_appointment_completion.py
```

## File Naming Conventions

- **Routers**: `[domain_name].py` (e.g., `appointment.py`, `auth.py`)
- **Services**: `[action]_[entity].py` (e.g., `create_appointment.py`, `get_user.py`)
- **Schemas**: `interfaces.py` (contains all request/response models for the domain)
- **Utils**: `[purpose]_utils.py` (e.g., `password_utils.py`, `datetime_utils.py`)

## Key Files

### Entry Points

- `main.py`: FastAPI app initialization, router registration, middleware setup
- `scripts.py`: CLI commands for running the server (`start`, `dev`, `prod`)

### Configuration

- `pyproject.toml`: uv dependencies, project metadata, tool configs
- `src/config.py`: Environment variables, Redis configuration
- `src/config_memory.py`: Memory optimization settings

### Shared Models

- `src/models.py`: Common Pydantic models used across domains

## Special Directories

### `docs/`

- Project documentation
- API documentation
- Setup guides

## Best Practices

1. **One Domain = One Module**: Each feature domain gets its own module directory
2. **Service Layer**: Business logic lives in services, not routers
3. **Schema Validation**: All request/response models in `schemas/interfaces.py`
4. **Shared Code**: Common functionality in `utils/` or `services/base_service.py`
5. **Consistent Structure**: All modules follow the same folder pattern

## Adding a New Domain Module

To add a new feature domain:

1. Create directory: `src/[new_domain]/`
2. Add `__init__.py`
3. Create router: `[new_domain].py` with FastAPI router
4. Create `schemas/interfaces.py` with Pydantic models
5. Create `services/` directory with business logic
6. Register router in `main.py`: `app.include_router([new_domain]_router, prefix="/[new-domain]", tags=["[new-domain]"])`
